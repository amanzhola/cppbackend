# 🚀 Restaurant (Async, Single Thread)

## Симуляция кухни через Boost.Asio без использования потоков и блокирующих операций

---

# 📌 Краткое описание

| 📌 Компонент | 📖 Что делает и как работает в рамках всей системы асинхронной обработки заказов                                                                         | ⚙️ Техническая реализация через Boost.Asio и внутреннюю архитектуру                                          | 🧠 Почему это важно для понимания асинхронных систем                              | 🔥 Что это демонстрирует на практике                      | ✅ Итог                                      |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------- | --------------------------------------------------------- | ------------------------------------------- |
| Restaurant   | Создаёт заказы, управляет их жизненным циклом и передаёт управление в асинхронную модель выполнения без блокировки основного потока выполнения программы | Использует io_context как центральный механизм планирования событий и создаёт объекты Order через shared_ptr | Позволяет понять, как централизованно управлять асинхронными задачами без потоков | Демонстрирует переход от линейного к событийному мышлению | Централизованная логика управления заказами |
| Order        | Выполняет заказ по этапам (жарка, маринование, упаковка) с контролем состояний и зависимостей между этапами                                              | Реализован как state machine с использованием таймеров и callback-функций                                    | Показывает, как строится асинхронный pipeline выполнения задач                    | Демонстрирует реальную бизнес-логику без блокировок       | Управляемый жизненный цикл заказа           |
| Timer        | Имитация длительных операций без блокировки потока выполнения                                                                                            | steady_timer с async_wait                                                                                    | Позволяет заменить sleep на неблокирующий механизм                                | Демонстрирует асинхронную модель времени                  | Основа всей системы                         |
| Logger       | Логирование этапов выполнения с точным временем и порядком событий                                                                                       | std::osyncstream для потокобезопасного вывода                                                                | Делает поведение системы прозрачным                                               | Показывает последовательность событий                     | Удобные читаемые логи                       |

# 🎯 Цель

| 🎯 Цель               | 📖 Подробное объяснение цели в контексте асинхронной архитектуры и event-driven подхода                               | 🔥 Что демонстрирует выполнение этой цели на практике         | ⚠️ Что произойдёт если это не понять и использовать неправильный подход | 🧠 Как это влияет на архитектуру  | ✅ Итог          |
| --------------------- | --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- | ----------------------------------------------------------------------- | --------------------------------- | --------------- |
| Понять async модель   | Вся система работает без блокировок, управление происходит через события и callbacks, а не через последовательный код | Можно обрабатывать несколько заказов одновременно без потоков | Использование sleep или блокировок разрушает модель                     | Формируется event-driven мышление | Понимание async |
| Понять параллельность | Операции выполняются логически параллельно внутри одного потока                                                       | Логи показывают пересечение этапов разных заказов             | Ошибка — думать, что нужен thread                                       | Формируется понимание concurrency | Async ≠ threads |
| Управление состоянием | Каждый этап проверяется и выполняется строго по условиям                                                              | Нет возможности выполнить этап раньше времени                 | Без проверок система ломается                                           | Вводится state machine            | Надёжность      |
| Работа с таймерами    | Таймеры управляют временем выполнения операций                                                                        | Нет блокировки CPU                                            | sleep блокирует систему                                                 | Лёгкая архитектура                | Async timing    |

# 🧠 Главная идея

| 🧠 Этап выполнения заказа и переходов между состояниями | 📖 Подробное описание того, что происходит внутри системы на этом этапе      | ⚙️ Как именно это реализовано через Boost.Asio и callbacks | 🔥 Что это демонстрирует в контексте асинхронного программирования | ⚠️ Типичная ошибка при реализации | ✅ Итог            |
| ------------------------------------------------------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------ | --------------------------------- | ----------------- |
| Старт заказа                                            | Метод Execute запускает процесс без блокировки и сразу возвращает управление | Просто вызывает RoastCutlet и возможно MarinadeOnion       | Нет ожидания выполнения                                            | Использование sleep               | Асинхронный старт |
| Жарка котлеты                                           | Таймер запускается и по завершению вызывает callback                         | async_wait                                                 | Код не блокируется                                                 | Blocking calls                    | Event-driven      |
| Маринование лука                                        | Параллельный таймер для второго этапа                                        | независимый timer                                          | независимые операции                                               | нарушение порядка                 | Параллельность    |
| Проверка готовности                                     | Проверяется можно ли продолжить выполнение                                   | CheckReadiness                                             | контроль состояния                                                 | race conditions                   | Надёжность        |
| Упаковка                                                | Финальный этап с имитацией работы                                            | busy loop                                                  | завершение заказа                                                  | ранний запуск                     | Завершение        |

# ⚠️ Ключевые ошибки

| ⚠️ Ошибка    | 📖 Почему это ломает архитектуру и нарушает асинхронную модель | 🔥 Что происходит в системе при этой ошибке | 🧠 Как правильно думать | ✅ Правильный подход |
| ------------ | -------------------------------------------------------------- | ------------------------------------------- | ----------------------- | ------------------- |
| sleep        | Блокирует поток                                                | вся система останавливается                 | думать event-driven     | использовать timer  |
| thread       | усложняет                                                      | лишняя синхронизация                        | async вместо thread     | io_context          |
| нет проверок | ломает порядок                                                 | invalid states                              | state machine           | guards              |
| ранний pack  | логика нарушена                                                | ошибка состояния                            | проверка условий        | CheckReadiness      |
