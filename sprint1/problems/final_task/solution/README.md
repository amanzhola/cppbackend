## 12. Правильная оптимизация кэша

### 12.1 Идея

| 📌 Идея оптимизации кэша Docker                             | 📖 Полное объяснение                                   | 🧠 Что это означает                             | ✅ Итог         |
| ----------------------------------------------------------- | ------------------------------------------------------ | ----------------------------------------------- | -------------- |
| Нужно разделить слои на редко меняющееся и часто меняющееся | `редко меняющееся -> выше`, `часто меняющееся -> ниже` | Это позволяет Docker использовать кэш правильно | Кэш ускоряется |

### 12.2 Оптимизированная версия

```dockerfile
FROM gcc:11.3

RUN apt update && \
    apt install -y \
      python3-pip \
      cmake \
    && \
    pip3 install conan==1.*

COPY conanfile.txt /app/
RUN mkdir /app/build && cd /app/build && \
    conan install .. --build=missing

COPY ./src /app/src
COPY ./data /app/data
COPY CMakeLists.txt /app/

RUN cd /app/build && \
    cmake -DCMAKE_BUILD_TYPE=Release .. && \
    cmake --build .
```

### 12.3 Что это даёт

| 📌 Результат оптимизации                                 | 📖 Полное объяснение                                                            | 🧠 Практический эффект                             | ✅ Итог                     |
| -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | -------------------------------------------------- | -------------------------- |
| Теперь `conan install` зависит только от `conanfile.txt` | Если меняется только код в `src`, Docker берёт слой с `conan install` из cache и не делает его заново | Сборка становится быстрее при изменении исходников | Кэш используется правильно |

| Промежуточный образ был слишком тяжёлым | Потому что внутри него были: `gcc`, `cmake`, `conan`, `python3-pip`, всё сборочное окружение. Но для запуска сервера это не нужно. ## 14. Финальный Dockerfile

```dockerfile
# Не просто создаём образ, но даём stage имя build
FROM gcc:11.3 as build

RUN apt update && \
    apt install -y \
      python3-pip \
      cmake \
    && \
    pip3 install conan==1.*

# Сначала зависимости
COPY conanfile.txt /app/
RUN mkdir /app/build && cd /app/build && \
    conan install .. --build=missing

# Потом исходники
COPY ./src /app/src
COPY CMakeLists.txt /app/

RUN cd /app/build && \
    cmake -DCMAKE_BUILD_TYPE=Release .. && \
    cmake --build .

# Второй stage: только запуск
FROM ubuntu:22.04 as run

# Создаём отдельного пользователя
RUN groupadd -r www && useradd -r -g www www
USER www

# Копируем собранный бинарник из stage build
COPY --from=build /app/build/bin/game_server /app/
COPY ./data /app/data

# Команда запуска контейнера
ENTRYPOINT ["/app/game_server", "/app/data/config.json"]
```

## 18. Объяснение разницы между первым и вторым запуском

### 18.1 Первый запуск — почему работал IP:8080

| 📌 Первый запуск контейнера                                                                        | 📖 Полное объяснение без потери содержания                                                                                                                                                                                                       | 🧠 Что это означало                                         | ✅ Итог                        |
| -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------- | ----------------------------- |
| Использовалась команда `sudo docker run --rm -it --entrypoint bash -p 8080:8080 game-server-image` | Это означало: `VPS:8080 -> container:8080`. Поэтому работали адреса: `http://94.131.92.240:8080/api/v1/maps` и `http://gameserver.hopto.org:8080/api/v1/maps`. Потому что Docker слушал именно порт 8080 на VPS и перенаправлял его в контейнер. | Внешний 8080 был открыт и реально проксировался в контейнер | IP:8080 и домен:8080 работали |

### 18.2 Второй запуск — почему IP:8080 перестал работать

| 📌 Второй запуск контейнера                                                | 📖 Полное объяснение без потери содержания                                                                                                                                                                                                                                                                                     | 🧠 Что изменилось                             | ✅ Итог                                        |
| -------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------- | --------------------------------------------- |
| Использовалась команда `sudo docker run --rm -p 80:8080 game-server-image` | Это означало уже другое правило: `VPS:80 -> container:8080`. То есть Docker больше не слушал порт `8080` на VPS. Поэтому адрес `http://94.131.92.240:8080/api/v1/maps` перестал работать. Браузер писал: `Сайт 94.131.92.240 не позволяет установить соединение`. По сути это означало: `на VPS:8080 сейчас никто не слушает`. | Слушающий внешний порт изменился с 8080 на 80 | Адрес с `:8080` закономерно перестал работать |

### 18.3 Почему домен без порта заработал

| 📌 Почему заработал адрес без `:8080`                                     | 📖 Полное объяснение без потери содержания                                                                                                                                                   | 🧠 Как это связано с HTTP по умолчанию            | ✅ Итог                                   |
| ------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- | ---------------------------------------- |
| Во втором запуске работал адрес `http://gameserver.hopto.org/api/v1/maps` | Потому что для HTTP адрес без указанного порта означает по умолчанию: `порт 80`. А во втором запуске как раз был проброс: `VPS:80 -> container:8080`. Поэтому всё было правильно и ожидаемо. | Браузер автоматически использовал внешний порт 80 | Домен без порта начал работать правильно |
