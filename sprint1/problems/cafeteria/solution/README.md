## Используемая архитектура

| 🧩 Компонент архитектуры | 📖 Полный текст объяснения без потери содержания                                                                                                                                                                                                                                                                     | 🧠 Архитектурный смысл                                                     | ✅ Итог                                  |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- | --------------------------------------- |
| `GasCooker`              | Готовая модель общей газовой плиты. Умеет: * асинхронно занимать горелку (`UseBurner`) * освобождать горелку (`ReleaseBurner`) Внутри использует `strand`, поэтому корректно работает при вызове из нескольких потоков.                                                                                              | Централизованная потокобезопасная модель плиты.                            | Безопасное управление общим ресурсом.   |
| `GasCookerLock`          | RAII-обёртка над занятой горелкой. RAII — **Resource Acquisition Is Initialization** Полностью: * **Resource** — ресурс * **Acquisition** — получение * **Is** — является * **Initialization** — инициализацией Смысл: * горелка автоматически освобождается при уничтожении объекта или вызове `Unlock()`.          | Автоматическое освобождение ресурса без ручных ошибок.                     | Безопасное владение горелкой.           |
| `Bread`                  | Реализован по аналогии с `Sausage`. Поддерживает: * `StartBake(...)` * `StopBaking()` * `IsCooked()` * `GetBakingDuration()` Логика полностью симметрична сосиске: * фиксируется фактическое время начала выпекания; * фиксируется время окончания; * хранится `GasCookerLock`, чтобы корректно освобождать горелку. | Симметричная модель второго ингредиента с теми же гарантиями корректности. | Корректная модель булки.                |
| `Cafeteria`              | Главный класс, который принимает заказы на хот-доги. Внутри использует: * `Store` — для получения ингредиентов; * `GasCooker` — общая плита; * `strand` — для сериализации доступа к общему состоянию кафетерия.                                                                                                     | Центральная точка оркестрации всей логики приготовления.                   | Координация заказов и защита состояния. |

## Как работает `OrderHotDog`

| 🧠 Шаг | 📖 Полный текст шага без потери содержания                                                                                                                                                                                                                                             | ⚙️ Что происходит в коде                                                  | ✅ Итог                                       |
| ------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | -------------------------------------------- |
| Шаг 1  | Вызов сериализуется через `strand` кафетерия. Это нужно для безопасного доступа к: * `Store` * счётчику id хот-догов                                                                                                                                                                   | Все входы в общую логику проходят последовательно.                        | Нет гонок на общем состоянии.                |
| Шаг 2  | Создаются: * булка * сосиска * уникальный id хот-дога                                                                                                                                                                                                                                  | Инициализация данных заказа.                                              | База для асинхронной цепочки готова.         |
| Шаг 3  | Создаётся объект состояния заказа (`OrderState`), который живёт до завершения всей асинхронной цепочки. Он хранит: * булку * сосиску * таймер булки * таймер сосиски * флаги готовности * обработчик результата * собственный `strand`                                                 | Всё состояние конкретного заказа изолируется в одном объекте.             | Жизненный цикл заказа контролируется.        |
| Шаг 4  | Булка и сосиска запускаются **параллельно**: * `bread->StartBake(...)` * `sausage->StartFry(...)` Каждый ингредиент: * сначала асинхронно занимает горелку; * после получения горелки запускает свой таймер; * по срабатыванию таймера завершает приготовление; * освобождает горелку. | Две независимые операции используют общую плиту через асинхронный доступ. | Реальный параллелизм на уровне ингредиентов. |
| Шаг 5  | Когда оба ингредиента готовы, вызывается сборка хот-дога: `HotDog{hotdog_id, sausage, bread}` Конструктор `HotDog` сам проверяет диапазоны допустимых длительностей. Если длительности не попадают в допустимые интервалы, выбрасывается исключение.                                   | Валидация происходит на этапе финальной сборки.                           | Некорректные хот-доги отсекаются.            |
| Шаг 6  | Результат передаётся в пользовательский `handler` через `Result<HotDog>`: * либо готовый хот-дог, * либо ошибка.                                                                                                                                                                       | Завершение заказа приходит наружу через callback.                         | Пользователь получает итог асинхронно.       |

## Почему решение потокобезопасно

| 🛡️ Механизм                       | 📖 Полный текст объяснения без потери содержания                                                                                                                                              | 🧠 Почему это работает                                                      | ✅ Итог                                       |
| ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | -------------------------------------------- |
| 1. Безопасность `GasCooker`        | `GasCooker` уже сам потокобезопасен, так как использует свой внутренний `strand`.                                                                                                             | Доступ к горелкам сериализуется внутри самой плиты.                         | Общий ресурс защищён.                        |
| 2. Безопасность `Cafeteria`        | В `Cafeteria` используется собственный `strand`, чтобы: * несколько потоков могли вызывать `OrderHotDog` одновременно, * но доступ к `Store` и счётчику хот-догов выполнялся последовательно. | Общие данные кафетерия не модифицируются конкурентно.                       | Нет гонок на глобальном состоянии кафетерия. |
| 3. Безопасность `OrderState`       | У каждого заказа свой отдельный объект состояния и свой `strand`. Это защищает внутреннее состояние заказа: * флаги готовности * таймеры * завершение заказа * сборку результата              | Внутренние этапы одного заказа не пересекаются опасно.                      | Заказ изолирован от других и от себя самого. |
| 4. Защита от повторного завершения | В `OrderState` используется флаг `completed`, чтобы завершение заказа происходило только один раз.                                                                                            | Даже при сложной асинхронной последовательности финализация не дублируется. | Нет двойного завершения.                     |

## Почему решение эффективно

| ⚡ Анализ эффективности                                                          | 📖 Полный текст объяснения без потери содержания                                                                                                                                     | 🧠 Вычислительный смысл                               | ✅ Итог                                     |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- | ------------------------------------------ |
| Хот-дог требует одновременно:                                                   | * 1 горелку для булки * 1 горелку для сосиски То есть один хот-дог во время приготовления занимает **2 горелки**.                                                                    | Один хот-дог = 2 ресурса плиты.                       | Ограничение по параллелизму понятно.       |
| Плита содержит **8 горелок**, значит одновременно можно эффективно готовить до: | `text 8 / 2 = 4 хот-дога `                                                                                                                                                           | Максимум 4 хот-дога параллельно.                      | Верхняя граница одновременной загрузки.    |
| Каждый хот-дог ограничен более длинной операцией — приготовлением сосиски:      | `text 1.5 секунды `                                                                                                                                                                  | Сосиска — бутылочное горлышко по времени.             | Именно она ограничивает throughput.        |
| Для 20 заказов это даёт примерно:                                               | * 4 хот-дога за 1.5 секунды * 20 хот-догов = 5 партий * 5 * 1.5 секунды ≈ 7.5 секунд На практике часть операций перекрывается эффективнее, и итог укладывается в требуемый интервал. | Теоретическая оценка времени согласуется с практикой. | Производительность соответствует ожиданию. |
