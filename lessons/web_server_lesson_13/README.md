# 1. Многопоточность через `RunWorkers`

```cpp
template <typename Fn>
void RunWorkers(unsigned n, const Fn& fn) {
    n = std::max(1u, n);
    std::vector<std::jthread> workers;
    workers.reserve(n - 1);

    while (--n) {
        workers.emplace_back(fn);
    }

    fn();
}
```

| 🔧 Элемент функции `RunWorkers` | 📖 Что он делает                              | 🧠 Почему это важно                                         | ⚠️ Что было бы без этого                             | ✅ Итог                              |
| ------------------------------- | --------------------------------------------- | ----------------------------------------------------------- | ---------------------------------------------------- | ----------------------------------- |
| `std::max(1u, n)`               | Не даёт создать ноль потоков исполнения       | Всегда будет хотя бы один исполнитель                       | Можно было бы случайно не запустить обработку вообще | Есть минимум один рабочий поток     |
| `workers.reserve(n - 1)`        | Резервирует память под фоновые потоки         | Уменьшает лишние перераспределения                          | Мелкая, но полезная оптимизация                      | Код аккуратнее                      |
| `workers.emplace_back(fn)`      | Создаёт `n - 1` фоновых потока                | Эти потоки будут выполнять одну и ту же рабочую функцию     | Без них всё работало бы только в одном потоке        | Получаем параллелизм                |
| `fn();` в текущем потоке        | Запускает ту же работу ещё и в текущем потоке | Текущий поток не простаивает, а тоже участвует в выполнении | Один поток был бы потерян зря                        | Итоговое число работников равно `n` |

# 4. Что такое `ThreadChecker`

```cpp
class ThreadChecker {
public:
    explicit ThreadChecker(std::atomic_int& counter)
        : counter_{counter} {
    }

    ThreadChecker(const ThreadChecker&) = delete;
    ThreadChecker& operator=(const ThreadChecker&) = delete;

    ~ThreadChecker() {
        assert(expected_counter_ == counter_);
    }

private:
    std::atomic_int& counter_;
    int expected_counter_ = ++counter_;
};
```

| 🧪 Часть `ThreadChecker`                | 📖 Что делает                                                              | 🧠 Для чего нужна                                      | 🔥 Что произойдёт при пересечении потоков                      | ✅ Итог                                     |
| --------------------------------------- | -------------------------------------------------------------------------- | ------------------------------------------------------ | -------------------------------------------------------------- | ------------------------------------------ |
| `std::atomic_int& counter_`             | Хранит ссылку на атомарный счётчик входов в критический участок            | Позволяет безопасно фиксировать пересечения потоков    | Обычная переменная сама была бы некорректной в многопоточности | Атомарность нужна для надёжной диагностики |
| `expected_counter_ = ++counter_`        | При создании объекта увеличивает счётчик и запоминает его значение         | Маркирует текущий вход в опасный участок               | Если другой поток войдёт туда же, счётчик изменится            | Можно поймать нарушение                    |
| `assert(expected_counter_ == counter_)` | При уничтожении проверяет, что счётчик не изменился за время жизни объекта | Обнаруживает вторжение другого потока в тот же участок | Программа аварийно завершится и покажет факт пересечения       | Проблема доказана                          |

# 5. Почему мьютекс — не лучший путь

```cpp
std::mutex mutex_;

void OnRoasted(sys::error_code ec) {
    std::lock_guard lk{mutex_};
    ...
}

void OnOnionMarinaded(sys::error_code ec) {
    std::lock_guard lk{mutex_};
    ...
}
```

| 🔒 Подход через `std::mutex`                  | 📖 Что он даёт                                             | ⚠️ Почему он не лучший для этого урока                         | 🧠 Что лучше в модели `Boost.Asio`                                               | ✅ Итог                                                            |
| --------------------------------------------- | ---------------------------------------------------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| Последовательный доступ к критическим секциям | Не даёт двум потокам одновременно выполнять опасные методы | Потоки начинают просто ждать друг друга вместо полезной работы | Лучше сериализовать обработчики через `strand`, а не блокировать поток мьютексом | Мьютекс возможен, но `strand` лучше вписывается в эту архитектуру |

# 6. Что такое `strand`

```cpp
net::strand<net::io_context::executor_type> strand_{net::make_strand(io_)};
```

```cpp
roast_timer_.async_wait(
    net::bind_executor(strand_, [self = shared_from_this()](sys::error_code ec) {
        self->OnRoasted(ec);
    }));

marinade_timer_.async_wait(
    net::bind_executor(strand_, [self = shared_from_this()](sys::error_code ec) {
        self->OnOnionMarinaded(ec);
    }));
```

| 🧵 Что даёт `strand`                                    | 📖 Как он работает                                                                      | 🧠 Что важно понять                                           | 🔥 Что сохраняется даже после его добавления              | ✅ Итог                                                     |
| ------------------------------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------- | --------------------------------------------------------- | ---------------------------------------------------------- |
| Последовательность обработчиков одной логической группы | Обработчики, привязанные к одному и тому же `strand`, не будут выполняться одновременно | Это не “один поток навсегда”, а именно гарантия непересечения | Разные `strand` по-прежнему могут выполняться параллельно | Безопасность внутри заказа и параллельность между заказами |

# Демо 3 — ресторан с обнаружением гонки

## Идея

Показать, что в многопоточном варианте у одного заказа действительно может возникнуть гонка, и `ThreadChecker` способен это обнаружить.

В этой версии:

* `io.run()` вызывается на 4 потоках;
* таймеры жарки и маринования уменьшены до `1ms`;
* в `OnRoasted` и `OnOnionMarinaded` используется `ThreadChecker`;
* внутрь обработчиков добавлена короткая пауза `sleep_for(2ms)`, чтобы увеличить вероятность пересечения.

## Запуск

./demo3_order_race
