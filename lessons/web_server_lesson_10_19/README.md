# 🚀 Практическое занятие №19

## Почему нельзя передавать `socket&` в поток

---

# 📌 Краткое описание

| 📌 Что изучается                                    | 📖 Полное описание                                                                                                                                                                                                                                                                                                                                                                |
| --------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Опасность передачи сокета по ссылке в `std::thread` | В этом занятии разбирается, почему передача клиентского `tcp::socket` в поток по ссылке через `socket&` и `std::ref(socket)` опасна, как это связано со временем жизни локальной переменной внутри цикла `while`, почему поток может обратиться к уже уничтоженному объекту и почему безопасный вариант состоит в передаче сокета по значению с последующим перемещением владения |

---

# 🎯 Цель

| 🎯 Что нужно понять                              | 📖 Что это означает на практике                                                                                                    |
| ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| Локальная переменная живёт ограниченное время    | Переменная `socket` существует только в пределах текущей итерации цикла, а после выхода из этой итерации объект уничтожается       |
| Поток может жить дольше, чем переменная в `main` | После `detach()` поток продолжает работу независимо и может попытаться использовать ссылку на объект, которого уже не существует   |
| Передача `socket&` опасна                        | Если поток хранит ссылку на локальный сокет из цикла, возникает риск обращения к уничтоженному объекту                             |
| Передача по значению безопасна                   | Если сокет перемещается в поток по значению, поток получает собственный объект и может безопасно работать с клиентским соединением |

---

# ⚙️ Основной код

```cpp
std::thread(
    [](tcp::socket socket) {
        HandleConnection(std::move(socket));
    },
    std::move(socket)
).detach();
```

---

# 🧠 Главная идея

| 🧠 Ключевая логика                                               | 📖 Смысл                                                                                                                                                                                                                                  |
| ---------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `accept → получить socket → переместить socket в поток → detach` | После принятия подключения главный поток не передаёт ссылку на локальную переменную, а отдаёт сам объект сокета новому потоку, благодаря чему время жизни соединения больше не зависит от времени жизни локальной переменной внутри цикла |

---

# ❌ Почему нельзя передавать `socket&`

| ❌ Опасная идея                                      | 📖 Почему это плохо                                                                                                                                           |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Передать `socket&` в поток через `std::ref(socket)` | Ссылка будет указывать на локальную переменную текущей итерации цикла, а поток после `detach()` может продолжить работу уже после уничтожения этой переменной |
| Дать потоку работать с внешним локальным объектом   | Возникает риск `dangling reference`, то есть обращения к объекту, который уже не существует                                                                   |
| Надеяться, что поток успеет раньше завершиться      | Такая логика ненадёжна, потому что порядок выполнения потоков не гарантируется и поведение зависит от планировщика                                            |

---

# ✅ Почему передача по значению безопасна

| ✅ Безопасный подход                            | 📖 Почему он работает правильно                                                                                                             |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Передать сокет в поток по значению             | Поток получает собственный объект `tcp::socket`, а не ссылку на чужую локальную переменную                                                  |
| Использовать `std::move(socket)`               | Владение соединением переносится в новый поток, и после этого главный поток уже не должен использовать старый объект как прежнего владельца |
| Обрабатывать клиента внутри `HandleConnection` | Весь жизненный цикл сокета теперь привязан к потоку, который обслуживает этого клиента                                                      |

---

# 🌐 Что происходит в программе

| 🌐 Шаг                                | 📖 Объяснение                                                                                                    |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `tcp::socket socket(ioc)`             | Внутри цикла создаётся локальная переменная сокета, которая принадлежит текущей итерации                         |
| `acceptor.accept(socket)`             | Сервер принимает клиентское подключение и связывает этот сокет с новым клиентом                                  |
| `std::thread(..., std::move(socket))` | Сокет передаётся в поток как перемещаемый объект, а не как ссылка на локальную переменную                        |
| `HandleConnection(std::move(socket))` | Поток получает владение соединением и независимо читает запросы и отправляет ответы                              |
| Конец итерации цикла в `main`         | Локальная переменная старой итерации больше не важна для обработки, потому что реальный сокет уже передан потоку |

---

# 🖥 Что могло бы пойти не так при передаче по ссылке

| 🖥 Проблемная ситуация                                     | 📖 Что это означает                                                                                              |
| ---------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Итерация цикла завершилась, а поток ещё работает           | Локальная переменная `socket` уже уничтожена, но поток всё ещё пытается использовать ссылку на неё               |
| Поток читает или пишет через ссылку на уничтоженный объект | Поведение становится неопределённым, возможны падения, ошибки и повреждение логики работы сервера                |
| Ошибка проявляется не всегда                               | Такие баги особенно опасны, потому что могут проявляться нестабильно и зависеть от времени выполнения и нагрузки |

---

# 🌐 HTTP смысл

| 🌐 Что происходит на уровне HTTP                                                              | 📖 Почему это важно                                                                                                                   |
| --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| Каждый клиент обслуживается через собственное TCP-соединение                                  | Поток, который читает HTTP-запрос и отправляет HTTP-ответ, должен владеть сокетом этого клиента                                       |
| Один клиент не должен зависеть от времени жизни локальной переменной в другом месте программы | Если обработка соединения идёт в отдельном потоке, ей нужен собственный объект сокета, а не ссылка на временную внешнюю переменную    |
| Безопасная модель владения делает сервер устойчивее                                           | Когда сокет принадлежит потоку обработки, клиентская сессия живёт ровно столько, сколько нужно для чтения запросов и отправки ответов |

# 🧪 Тесты

| 🧪 Что выполнить                                                                                   | ✅ Что должно получиться                                                                                            |
| -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Выполнить обычный запрос `curl http://localhost:8080/`                                             | Сервер должен вернуть корректный HTTP-ответ с телом, сформированным в обработчике                                  |
| Выполнить несколько запросов подряд                                                                | Сервер должен продолжать работать стабильно и принимать повторные подключения                                      |
| Выполнить несколько запросов одновременно через фоновые процессы                                   | Все клиенты должны получать ответы, а сервер не должен переставать принимать новые подключения                     |
| Выполнить нагрузочный сценарий с большим числом фоновых запросов, а затем отдельный обычный запрос | Даже при нагрузке сервер должен продолжать отвечать новому клиенту, потому что соединения обслуживаются независимо |
