# 📌 Краткое описание

| -------------------- | ------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- | -------------------------------- | -------------------------------- |
| Hamburger            | Представляет состояние одного гамбургера и хранит информацию о том, прожарена ли котлета, добавлен ли лук и упакован ли продукт | Обычный класс с полями bool и методами проверки и изменения состояния                           | Это чистая бизнес-модель без async и без сетевой логики           | Разделение логики и состояния    | Чистая модель предметной области |

# 🏗 Полная структура программы в виде дерева

```text
Программа
├── class Hamburger
│   ├── состояние гамбургера
│   ├── методы изменения состояния
│   └── проверки корректности
│
├── class Logger
│   ├── id логгера
│   └── синхронизированный вывод сообщений
│
├── class Order : public std::enable_shared_from_this<Order>
│   ├── хранит один заказ
│   ├── знает, нужен ли лук
│   ├── хранит Hamburger
│   ├── хранит таймер жарки
│   ├── хранит таймер маринования
│   ├── запускает асинхронные операции
│   ├── получает события их завершения
│   ├── проверяет готовность
│   ├── упаковывает
│   └── доставляет клиенту результат
│
├── class Restaurant
│   ├── хранит io_context
│   ├── выдаёт id заказам
│   └── создаёт Order
│
├── operator<<(std::ostream&, const Hamburger&)
│   └── печатает гамбургер
│
└── main()
    ├── создаёт io_context
    ├── создаёт Restaurant
    ├── создаёт заказы
    └── запускает io.run()
```

# 🧠 Как реализована система

| 🧠 Этап выполнения заказа и переходов между состояниями | 📖 Подробное описание того, что происходит внутри системы при выполнении этого этапа и какие изменения состояния происходят | ⚙️ Как это реализовано через Boost.Asio, таймеры и callback-функции с использованием `shared_from_this` | 🔥 Что это демонстрирует с точки зрения асинхронной архитектуры и event-driven модели выполнения | ⚠️ Типичная ошибка при неправильной реализации такого этапа | ✅ Итог          |
| ------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ | ----------------------------------------------------------- | --------------- |
| Execute                                                 | Запускает заказ, стартует все операции без ожидания их завершения и сразу возвращает управление вызывающему коду            | Вызывает `RoastCutlet()` и при необходимости `MarinadeOnion()` без блокировки                           | Нет блокировки и мгновенный возврат управления                                                   | Использование sleep или ожидания                            | Async старт     |
| RoastCutlet                                             | Запускает таймер на прожарку котлеты и регистрирует callback, который будет вызван после завершения                         | `steady_timer`, `expires_after`, `async_wait`, lambda с `shared_from_this()`                            | Обработка через события                                                                          | Потеря объекта `Order` до конца операции                    | Event-driven    |
| MarinadeOnion                                           | Параллельно запускает процесс маринования лука, если он нужен                                                               | Второй таймер с отдельным callback                                                                      | Независимые операции                                                                             | Жёсткая последовательность вместо async                     | Concurrency     |
| OnRoasted                                               | Обрабатывает завершение прожарки, пишет лог и обновляет состояние гамбургера                                                | Вызывается из callback таймера жарки                                                                    | Реакция на событие                                                                               | Нет обработки ошибок                                        | State update    |
| OnOnionMarinaded                                        | Обрабатывает завершение маринования, пишет лог и обновляет флаг `onion_marinaded_`                                          | Вызывается из callback таймера лука                                                                     | Независимые события сходятся в одном объекте                                                     | Нет контроля состояния                                      | Async events    |
| CheckReadiness                                          | Проверяет, можно ли продолжить выполнение и перейти к следующему этапу                                                      | Логика проверки состояний и условий                                                                     | Управление переходами в машине состояний                                                         | Отсутствие единой точки принятия решения                    | Контроль        |
| Pack                                                    | Выполняет финальную упаковку после готовности всех компонентов                                                              | Имитация работы + изменение состояния гамбургера + лог                                                  | Финальный этап                                                                                   | Ранний вызов упаковки                                       | Завершение      |
| Deliver                                                 | Завершает заказ и вызывает пользовательский handler                                                                         | Передаёт наружу результат готового заказа                                                               | Точка завершения жизненного цикла                                                                | Двойная доставка результата                                 | Result delivery |

# ⚠️ Ключевая особенность: `shared_from_this`

| ⚠️ Проблема                                       | 📖 Что происходит в асинхронном коде                               | 🔥 Почему это опасно                                | 🧠 Как решается правильно                                                                  | ✅ Итог                         |
| ------------------------------------------------- | ------------------------------------------------------------------ | --------------------------------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------ |
| Захват сырого `this` в callback                   | Таймер завершится позже, а объект `Order` может уже быть уничтожен | Возникает use-after-free и неопределённое поведение | Наследование от `std::enable_shared_from_this<Order>` и захват `self = shared_from_this()` | Безопасное время жизни объекта |
| Создание `shared_ptr(this)` вручную               | Появляется второй независимый владелец одного и того же объекта    | Возможен double delete                              | Использовать только `shared_from_this()`                                                   | Правильное владение            |
| Асинхронные операции живут дольше текущего вызова | callback будет вызван в будущем, а не прямо сейчас                 | Локальная логика может “закончиться” раньше объекта | Делать объект совместно-владеемым через `shared_ptr`                                       | Lifetime-safe async            |

# 🌐 Архитектурный смысл

| 🌐 Концепция  | 📖 Что это означает для общей модели программы                                         | 🧠 Как реализовано в этом уроке                 | 🔥 Почему это важно                   | ✅ Итог             |
| ------------- | -------------------------------------------------------------------------------------- | ----------------------------------------------- | ------------------------------------- | ------------------ |
| Event loop    | Все события выполняются через единый цикл обработки                                    | `io_context.run()`                              | Это основа всей async модели          | Async foundation   |
| State machine | Объект `Order` решает, в каком состоянии сейчас заказ и что делать дальше              | `CheckReadiness()` и флаги состояния            | Нельзя просто писать линейный код     | Контроль логики    |
| Composition   | Архитектура строится не на глубоком наследовании, а на включении объектов друг в друга | `Order` содержит `Hamburger`, `Logger`, `Timer` | Гибкий и понятный дизайн              | Хорошая композиция |
| Async flow    | Этапы работают через события завершения, а не через блокирующее ожидание               | timers + callbacks                              | Это ближе к реальным backend-системам | Production mindset |

# ✅ Формулировка проверки

| ✅ Что нужно подтвердить              | 📖 Как это видно в коде и логике                                | 🔥 Что считается ошибкой                                 | 🏁 Что считается успехом                 |
