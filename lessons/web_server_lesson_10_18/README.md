# 🚀 Практическое занятие №9

## Два клиента и отсутствие блокировки сервера

---

# 📌 Кратко

| 📌 Что это                                                        | 📖 Полное описание                                                                                                                                                                                                                                                                        |
| ----------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Мысленный эксперимент с двумя браузерами и многопоточным сервером | В этом задании нужно представить ситуацию, когда браузер A уже подключился и удерживает соединение открытым, а браузер B пытается подключиться позже, после чего сравнить поведение старого однопоточного сервера и нового многопоточного варианта с отдельным потоком на каждого клиента |

---

# 🎯 Цель

| 🎯 Что нужно понять                                           | 📖 Объяснение                                                                                                        |
| ------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Один клиент не должен блокировать всех остальных              | Если один клиент удерживает соединение или обрабатывается долго, это не должно останавливать приём новых подключений |
| Главный поток должен продолжать выполнять accept              | После передачи клиента в отдельный поток сервер должен немедленно вернуться к ожиданию следующего подключения        |
| Многопоточность устраняет блокировку приёма новых подключений | Новый клиент не должен ждать завершения старого клиента, если архитектура сервера построена правильно                |

---

# 🧠 Главная идея

| 🧠 Ключевая логика                                                                                                                                                              | 📖 Смысл                                                                                                                                                                        |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `браузер A удерживает соединение, но работает в отдельном потоке, а главный поток уже свободен и может принять браузер B, после чего для него тоже запускается отдельный поток` | Многопоточность убирает главную проблему однопоточного сервера: старый клиент больше не блокирует приём новых клиентов и не удерживает главный цикл сервера внутри одной сессии |

---

# 🌐 Сценарий

| 🌐 Ситуация                                        | 📖 Объяснение                                             |
| -------------------------------------------------- | --------------------------------------------------------- |
| Браузер A подключился первым                       | Сервер уже работает с одним клиентом                      |
| Браузер A удерживает соединение                    | Нужно понять, мешает ли это новым подключениям            |
| Браузер B подключается позже                       | Нужно проверить, будет ли он ждать или будет принят сразу |
| Сервер должен показать разницу между архитектурами | Именно это и является главным содержанием задания         |

---

# ❌ Поведение старого однопоточного сервера

| ❌ Что происходило раньше                                                                                                | 📖 Почему это было плохо                                                                                                                                                    |
| ----------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Сервер принимал браузер A и входил в обработку его соединения в том же самом потоке, который отвечает за приём клиентов | Пока браузер A удерживал соединение, сервер не возвращался к `accept`, а значит браузер B не мог быть принят немедленно и был вынужден ждать окончания работы с браузером A |
| Один клиент фактически удерживал главный поток сервера                                                                  | Из-за этого старый сервер вёл себя как одноклиентский, даже если логически предполагалось, что клиентов может быть много                                                    |

---

# ✅ Поведение нового многопоточного сервера

| ✅ Что происходит теперь                                                                                       | 📖 Почему это хорошо                                                                                                                   |
| ------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Сервер принимает браузер A, создаёт для него отдельный поток и сразу возвращается к `accept` в главном потоке | Даже если браузер A удерживает соединение, главный поток больше не занят его обработкой и остаётся доступным для приёма новых клиентов |
| Когда позже приходит браузер B, главный поток уже свободен и принимает его сразу                              | Для браузера B создаётся ещё один отдельный поток, поэтому он не зависит от состояния браузера A и не ждёт завершения его сессии       |

---

# 🔍 Проверка и результат

| 🔍 Что выполнить                                                                                                                     | 📥 Какой результат должен быть                                                                                                                                                              |
| ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Запустить сильную нагрузку множеством клиентов A и во время этой нагрузки выполнить отдельный запрос клиентом B из другого терминала | Несмотря на то что сервер уже занят обработкой множества подключений, отдельный клиент B всё равно получает ответ, а сервер продолжает принимать новые соединения без глобальной блокировки |

---

# 🧪 Клиент A, создающий нагрузку

```bash
for i in {1..1000}; do
  curl -s http://localhost:8080/ &
done
wait
```

---

# 🧪 Клиент B, приходящий позже

```bash
curl http://localhost:8080/
```

---

# 🔍 Как выполнять проверку правильно

| 🔍 Шаг проверки                                                                                        | 📥 Ожидаемый результат                                                                                                                        |
| ------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Сначала запустить клиента A, который создаёт большую параллельную нагрузку множеством фоновых запросов | Сервер получает много одновременных подключений и начинает их обрабатывать                                                                    |
| Пока нагрузка ещё идёт, во втором терминале запустить клиента B как отдельный обычный `curl` запрос    | Клиент B тоже получает ответ и не ждёт окончания всей предыдущей нагрузки, потому что главный поток сервера не перестал принимать подключения |

---

# 🧠 Что это доказывает

| 🧠 Сценарий                                                                            | 📖 Вывод                                                                                                     |
| -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| Большое количество клиентов A уже подключено и создаёт существенную нагрузку на сервер | Сервер загружен, но это не превращает его обратно в однопоточный блокирующий обработчик одного клиента       |
| Новый клиент B приходит уже во время этой нагрузки и делает свой отдельный запрос      | Новый клиент не блокируется старыми клиентами и получает возможность быть принятым и обработанным независимо |
| Главный поток продолжает работать даже под нагрузкой                                   | Именно это и доказывает, что исчезла блокировка приёма новых подключений из-за обработки старых соединений   |

---

# 🖥 Что видно в логах сервера

| 🖥 Что видно в терминале сервера                                                                                                                                              | 📖 Что это доказывает                                                                                                                                                      |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| В логах появляются многочисленные запросы от клиентов A, а затем среди них появляется ещё один новый запрос от клиента B, и сервер при этом не перестаёт принимать соединения | Сервер не застрял на одном клиенте и не перестал выполнять `accept`, а значит архитектура с отдельным потоком на каждого клиента действительно устраняет старую блокировку |

---

# 🌐 Архитектурный смысл задания

| 🌐 Что происходит на уровне архитектуры                                                                                                       | 📖 Почему это важно                                                                                            |
| --------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| Главный поток сервера отвечает только за принятие новых подключений и не должен надолго удерживаться внутри обработки одной клиентской сессии | Если главный поток перестаёт делать `accept`, новый клиент фактически теряет возможность быть принятым вовремя |
| Поток клиента A обслуживает только клиента A, а поток клиента B обслуживает только клиента B                                                  | Это разделяет нагрузку между клиентами и делает сервер по-настоящему многоклиентским                           |

# 🧪 Полная инструкция проверки

| 📋 Что сделать                                                                                                                                                                                                                               | ✅ Как понять, что всё правильно                                                                                                                                                                                                                                   |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Запустить сервер, затем в одном терминале создать сильную нагрузку множеством параллельных клиентов A, а во втором терминале во время этой нагрузки выполнить отдельный запрос клиентом B и посмотреть и на ответ клиента, и на логи сервера | Если клиент B не ждёт завершения всех клиентов A, получает ответ во время нагрузки, а сервер продолжает принимать новые подключения и выводить новые запросы в лог, значит цель упражнения достигнута и многопоточность действительно устранила старую блокировку |

# ✅ Формулировка проверки

| ✅ Проверка                                                                                                                                                                           | ✅ Условие выполнено                                                                                                                                                |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Представить или проверить сценарий, в котором браузер A удерживает соединение, а браузер B подключается позже, и сравнить старый однопоточный сервер с новым многопоточным вариантом | Если становится ясно, что в старом варианте браузер B ждал бы, а в новом варианте он принимается сразу и получает собственный поток, то цель упражнения достигнута |

# ⚠️ Что именно исчезает после изменения архитектуры

| ⚠️ Что было плохо раньше                                                                                                                     | 📖 Что исчезает теперь                                                                                                                                                                        |
| -------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Обработка старого клиента могла удерживать главный поток сервера и тем самым блокировать приём новых подключений до завершения старой сессии | Исчезает блокировка приёма новых подключений из-за обработки старого клиента, потому что обслуживание каждого клиента вынесено в отдельный поток и больше не удерживает главный цикл `accept` |

---

# 🏗 Архитектура

| 🏗 Компоненты                                                                         | 📖 Роль каждого компонента                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `main`, `acceptor.accept`, `std::thread`, `HandleConnection`, `detach`, `tcp::socket` | `main` управляет жизнью процесса и принимает подключения, `acceptor.accept` получает нового клиента, `std::thread` создаёт отдельный поток обработки, `HandleConnection` обслуживает конкретную клиентскую сессию, `detach` позволяет главному потоку сразу вернуться к `accept`, а `tcp::socket` переносит соединение в поток обслуживания |

| 🏁 Что доказано этой работой                                                                                                                                                                                                      | 📌 Вывод                                                                                                                                                                                                 |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| После переноса обработки клиента в отдельный поток сервер перестаёт блокировать приём новых подключений из-за старого клиента, поэтому браузер B может быть принят даже тогда, когда браузер A всё ещё удерживает своё соединение | Это важный шаг к пониманию практического backend-сервера: многопоточность убирает глобальную блокировку одного клиента и позволяет серверу оставаться доступным для новых подключений даже под нагрузкой |
