| 🧠 Ключевая концепция | 📖 Полное объяснение без потери содержания                                                              | ⚙️ Как это проявляется в коде                                               | ✅ Итог                              |
| --------------------- | ------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | ----------------------------------- |
| HTTP-сессия           | HTTP-сессия в этом уроке — это непрерывный сеанс обмена данными между клиентом и сервером               | Это конкретное TCP/HTTP-взаимодействие между сокетом клиента и сервером     | Это серверная сессия обмена данными |
| Web-сессия            | Web-сессия — это механизм идентификации клиента. Это другое понятие                                     | Не нужно путать session в сетевом смысле с web-session уровня cookies/login | Термины различаются                 |

| Graceful shutdown     | Сервер должен корректно останавливаться по `SIGINT` и `SIGTERM`, а не падать грубо                      | Для этого используется `signal_set` и `ioc.stop()`                          | Shutdown реализуется управляемо     |

## Основная архитектура

| 🏗 Компонент архитектуры  | 📖 Полное описание его роли в системе                                                                                         | 🧠 Почему он выделен в отдельный компонент                                                            | ✅ Итог                            |
| ------------------------- | ----------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | --------------------------------- |
| `Listener`                | Принимает входящие TCP-соединения и создаёт `Session`                                                                         | Listener не должен заниматься HTTP-логикой, он только принимает клиентов и передаёт соединение дальше | Разделение обязанностей           |
| `SessionBase`             | Хранит общую инфраструктуру сессии: `beast::tcp_stream`, `beast::flat_buffer`, `HttpRequest` и реализует общий цикл обработки | Всё, что не зависит от типа обработчика запроса, живёт в базовом классе и не дублируется              | Общая логика вынесена в базу      |
| `Session<RequestHandler>` | Хранит `RequestHandler` и делегирует ему обработку HTTP-запроса                                                               | Шаблонность нужна, потому что обработчик может быть функцией, лямбдой или функтором                   | Обработка запроса остаётся гибкой |
| `ServeHttp(...)`          | Прячет сложность серверной инфраструктуры и позволяет запустить сервер, просто передав обработчик запросов                    | Пользователь не должен вручную собирать Listener/Session/lifetime management снаружи                  | Получается удобный фасад          |

## Общий цикл обработки session

```text
Run → Read → OnRead → HandleRequest → Write → OnWrite → Read
```

| 🔄 Этап жизненного цикла одной HTTP-сессии | 📖 Что происходит на этом этапе                                                 | 🧠 Зачем этот этап нужен                                                     | ✅ Итог                                   |
| ------------------------------------------ | ------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- | ---------------------------------------- |
| `Run`                                      | Запускает жизненный цикл сессии и переносит дальнейшую работу в executor сокета | Нужно быстро выйти из strand акцептора и не задерживать приём новых клиентов | Session стартует правильно               |
| `Read`                                     | Асинхронно читает HTTP-запрос через `http::async_read`                          | Без этого сервер не получит следующий запрос клиента                         | Начинается чтение                        |
| `OnRead`                                   | Обрабатывает закрытие соединения клиентом, ошибки чтения или успешный запрос    | Это развилка после чтения                                                    | Управление переходит дальше              |
| `HandleRequest`                            | Делегирует обработку пользовательскому request handler                          | Серверное ядро отделено от прикладной логики                                 | Обрабатывается маршрут                   |
| `Write`                                    | Асинхронно отправляет HTTP-ответ через `http::async_write`                      | Ответ отправляется клиенту неблокирующе                                      | Начинается запись                        |
| `OnWrite`                                  | Решает: закрыть соединение или читать следующий запрос                          | Здесь поддерживается keep-alive                                              | Цикл либо завершается, либо продолжается |

## Подробный разбор архитектуры

### 8.1 Listener

| 🧩 Элемент `Listener` | 📖 Полное описание без потери содержания                                                                                                                                                                          | 🧠 Почему это хорошо                                                                         | ✅ Итог                         |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- | ------------------------------ |
| Listener              | Listener: слушает порт; принимает новое TCP-соединение; создаёт Session; сразу снова начинает ждать следующее соединение. Listener не занимается HTTP. Он только: принимает клиентов; передаёт соединение дальше. | Это разделение обязанностей: listener занимается только accept-логикой, а не обработкой HTTP | Архитектура чище и расширяемее |

### 8.2 SessionBase

| 🧩 Элемент `SessionBase`                       | 📖 Полное описание без потери содержания                                                                                                        | 🧠 Почему это важно                              | ✅ Итог                              |
| ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ | ----------------------------------- |
| Что хранит `SessionBase`                       | SessionBase хранит всё, что не зависит от типа обработчика: `beast::tcp_stream stream_`, `beast::flat_buffer buffer_`, `HttpRequest request_`   | Это общая инфраструктура любой session           | База сессии вынесена отдельно       |
| Почему `tcp_stream`, а не просто `tcp::socket` | Потому что `beast::tcp_stream` поддерживает: таймауты чтения; таймауты записи                                                                   | Это делает session более практичной и безопасной | Выбран более сильный тип            |
| Почему `flat_buffer`                           | Потому что HTTP-запрос нужно читать в буфер, а не просто “в одну строку”                                                                        | HTTP-чтение требует отдельного буфера протокола  | Буфер нужен обязательно             |
| Почему `request_` — поле класса                | Потому что в асинхронном коде локальные переменные функции не переживут цепочку операций: `Read`, `OnRead`, `HandleRequest`, `Write`, `OnWrite` | Данные запроса должны жить дольше одной функции  | request_ хранится в объекте session |

### 8.3 Session<RequestHandler>

| 🧩 Элемент `Session<RequestHandler>`       | 📖 Полное описание без потери содержания                                                      | 🧠 Почему это важно                                        | ✅ Итог                 |
| ------------------------------------------ | --------------------------------------------------------------------------------------------- | ---------------------------------------------------------- | ---------------------- |
| Что хранит шаблонный класс                 | Это шаблонный класс, который хранит только: `RequestHandler request_handler_;`                | Обработчик запроса — единственная шаблонно-зависимая часть | Шаблонность минимальна |
| Почему шаблонный                           | Потому что тип обработчика запроса может быть разным: функция, лямбда, объект-функтор         | Нельзя жёстко зашить один тип handler                      | Гибкость интерфейса    |
| Почему `SessionBase` и `Session` разделены | Чтобы код, не зависящий от `RequestHandler`, не дублировался для каждой специализации шаблона | Это уменьшает дублирование и упрощает поддержку            | Архитектура чище       |

### 8.4 Run()

| 🧩 Метод `Run()` | 📖 Полное описание без потери содержания                                                                                                                                                                                                                                                                                                                                                                 | 🧠 Почему он именно в базе                                                 | ✅ Итог                               |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- | ------------------------------------ |
| `void Run()`     | Метод `void Run()` запускает жизненный цикл сессии. Почему он в `SessionBase`: потому что логика запуска сессии не зависит от типа `RequestHandler`. Что он делает: он вызывает `net::dispatch(stream_.get_executor(), ...)`, то есть переносит дальнейшую работу в executor, связанный с конкретным `stream_`. Это важно, чтобы: быстро выйти из strand акцептора; не задерживать приём новых клиентов. | Это ускоряет accept-loop и правильно переводит обработку в контекст сокета | Session стартует через свой executor |

### 8.5 Read()

| 🧩 Метод `Read()` | 📖 Полное описание без потери содержания                                                                                                                                           | 🧠 Почему это важно                                                                                                        | ✅ Итог                                    |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| `Read()`          | Метод `Read()`: очищает старый запрос: `request_ = {};`; ставит таймаут: `stream_.expires_after(std::chrono::seconds(30));`; запускает асинхронное чтение: `http::async_read(...)` | Если соединение keep-alive, в рамках одной сессии может прийти несколько запросов, поэтому старый `request_` нужно очищать | Read готовит session к следующему запросу |

### 8.6 OnRead()

| 🧩 Метод `OnRead()`              | 📖 Полное описание трёх сценариев без потери содержания                                                                                                                                                                                     | 🧠 Что это означает                                        | ✅ Итог                          |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------- |
| После чтения возможны 3 ситуации | 1. Клиент закрыл соединение: `if (ec == http::error::end_of_stream) return Close();` Это нормальная ситуация. 2. Ошибка чтения: `if (ec) return ReportError(ec, "read");` 3. Успешно прочитан запрос: `HandleRequest(std::move(request_));` | `OnRead()` — это точка ветвления между close/error/success | Чтение корректно обрабатывается |

### 8.7 HandleRequest()

| 🧩 Метод `HandleRequest()`                    | 📖 Полное описание без потери содержания                                                                                                                                                                                                                                               | 🧠 Почему sender сделан именно так                                                                                                      | ✅ Итог                                  |
| --------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| `HandleRequest()` в `Session<RequestHandler>` | Это уже метод `Session<RequestHandler>`. Он вызывает пользовательский `request_handler_` и передаёт ему: HTTP-запрос; sender-функцию. Вот это место: `request_handler_(std::move(request), [self = this->shared_from_this()](auto&& response) { self->Write(std::move(response)); });` | sender — generic lambda, потому что позже обработчик может отдавать разные типы ответов. `auto&& response` делает sender универсальным. | Request handler отделён от session ядра |

### 8.8 Write()

| 🧩 Метод `Write()` | 📖 Полное описание без потери содержания                                                                                                                                                                                                                                                                                                                                                           | 🧠 Почему нужен `shared_ptr`                                                          | ✅ Итог                           |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- | -------------------------------- |
| `Write()`          | Метод `Write()` асинхронно отправляет HTTP-ответ. Важная проблема: ответ нельзя держать как локальную переменную на стеке до конца `async_write`, потому что `async_write` завершится позже. Решение: переместить ответ в кучу: `auto safe_response = std::make_shared<http::response<Body, Fields>>(std::move(response));` и захватить `safe_response` в лямбде. Так ответ живёт до конца записи. | В асинхронном коде lifetime ответа должен пережить момент выхода из функции `Write()` | Ответ живёт до завершения записи |

### 8.9 OnWrite()

| 🧩 Метод `OnWrite()`         | 📖 Полное описание трёх сценариев без потери содержания                                                                                                                                                                    | 🧠 Что это означает для keep-alive                      | ✅ Итог                                         |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- | ---------------------------------------------- |
| После записи есть 3 сценария | 1. Ошибка: `if (ec) return ReportError(ec, "write");` 2. Ответ требует закрытия соединения: `if (close) return Close();` 3. Можно продолжать: `Read();` То есть начинается чтение следующего HTTP-запроса в той же сессии. | Это и есть поддержка persistent connection / keep-alive | Session либо закрывается, либо продолжает цикл |

### 8.10 Цикл сессии

| 🔄 Полная цепочка и объяснение, почему это не рекурсия                               | 📖 Полный текст без потери содержания                                                                                                                                                                                                   | 🧠 Практический смысл                                               | ✅ Итог                                     |
| ------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------------ |
| Полная цепочка такая: `Run ↓ Read ↓ OnRead ↓ HandleRequest ↓ Write ↓ OnWrite ↓ Read` | Почему это не рекурсия: потому что `OnRead` и `OnWrite` вызываются асинхронно: не напрямую из функции; а позже, когда операция чтения или записи завершится. Это событийный цикл, а не обычная рекурсия. Это прямо объясняется в уроке. | Никакого бесконечного синхронного стека нет, есть событийная модель | Цикл session асинхронный, а не рекурсивный |

### 8.11 ServeHttp(...)

| 🧩 Фасадная функция `ServeHttp(...)`                     | 📖 Полное описание без потери содержания                                                                                                                                                                                                                                                                                    | 🧠 Что даёт `std::decay_t<RequestHandler>`                                           | ✅ Итог                             |
| -------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------- |
| `template <typename RequestHandler> void ServeHttp(...)` | Это удобная фасадная функция. Что делает `std::decay_t<RequestHandler>`: удаляет ссылки, const, volatile, чтобы Listener хранил обработчик по значению. Что делает `ServeHttp`: скрывает всю сложность: Listener, Session, Read/Write, управление временем жизни. Снаружи пользователь просто передаёт обработчик запросов. | Обработчик нормализуется и безопасно хранится внутри listener/session инфраструктуры | Снаружи интерфейс остаётся простым |

## Подробный разбор main.cpp

### 9.1 HandleRequest

| 🧩 Обработчик приложения | 📖 Полное описание без потери содержания                                                                                                                                                                                         | 🧠 Что именно он делает                                                     | ✅ Итог                                 |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | -------------------------------------- |
| `HandleRequest`          | Это обработчик приложения. Что он делает: принимает HTTP-запрос; возвращает HTTP-ответ. Поддержаны маршруты: `/`, `/ping`, `/hello`. И ошибки: любой метод кроме GET → 405 Method Not Allowed; неизвестный путь → 404 Not Found. | Это и есть пользовательская бизнес-логика приложения поверх серверного ядра | Маршрутизация реализована в приложении |

### 9.2 make_text_response

| 🧩 Вложенная лямбда `make_text_response` | 📖 Полное описание без потери содержания                                                                                                                            | 🧠 Почему `keep_alive` важно                                                                                                                           | ✅ Итог                                                     |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------- |
| `make_text_response`                     | Вложенная лямбда создаёт текстовый ответ. Что выставляется: статус, версия HTTP, `Content-Type: text/plain`, тело, `Content-Length`, `keep_alive(req.keep_alive())` | Потому что сервер должен уважать семантику соединения клиента: если клиент хочет keep-alive, сервер не обязан закрывать соединение после одного ответа | Ответ формируется корректно и уважает семантику соединения |

### 9.3 signal_set

| 🧩 Корректное завершение в main | 📖 Полное описание без потери содержания                                                          | 🧠 Связь с предыдущим уроком                                        | ✅ Итог                                   |
| ------------------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- | ---------------------------------------- |
| `signal_set` в main             | В `main` снова есть корректное завершение по: `SIGINT`, `SIGTERM`. Это продолжение темы урока 17. | Сервер не только умеет обслуживать HTTP, но и правильно завершаться | Урок 18 продолжает lesson 17 по shutdown |

### 9.4 ServeHttp

| 🧩 Где сервер реально запускается | 📖 Полное описание без потери содержания                                                                                                                                                                                                                                                                                                              | 🧠 Что означает эта лямбда                                                                         | ✅ Итог                                 |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- | -------------------------------------- |
| Запуск через `ServeHttp`          | Вот здесь сервер реально запускается: `http_server::ServeHttp(ioc, {address, port}, [](auto&& req, auto&& sender) { sender(HandleRequest(std::forward<decltype(req)>(req))); });` Это значит: сервер принимает соединения; создаёт Session; читает запрос; потом вызывает эту лямбду; лямбда вызывает HandleRequest; и отправляет ответ через sender. | Пользовательский код сводится к обработчику запроса, а всё остальное скрыто внутри серверного ядра | Сервер запускается через удобный фасад |

| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | ----------------------------------------------------- |
| 👉 то есть без `\n`              | Не нужно добавлять `\n` в тело ответа только ради красивого отображения в терминале, если это меняет смысл или длину тела ответа. Вариант 3 (хитрый): не менять сервер, а сделать в curl: `curl -i http://127.0.0.1:8080/Ivan && echo` или: `curl http://127.0.0.1:8080/Ivan; echo` | Это позволяет красиво увидеть вывод в терминале, не меняя сам серверный ответ | Красивый вывод делается на стороне curl, а не сервера |

## Что реализовано в SessionBase

| 🧩 Метод или идея | 📖 Полное описание без потери содержания                                    | ✅ Итог                                 |
| ----------------- | --------------------------------------------------------------------------- | -------------------------------------- |
| `Run()`           | Запускает жизненный цикл сессии и переносит работу в executor сокета.       | Session стартует в правильном executor |

| `OnWrite()`       | Решает: закрыть соединение; или читать следующий запрос.                    | Поддержка keep-alive                   |

## Почему response переносится в shared_ptr

| 🧠 Lifetime ответа при async_write | 📖 Полное объяснение без потери содержания                                                                                                                                                                              | ✅ Итог                                  |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| Почему нужен `shared_ptr`          | Запись выполняется асинхронно. Если держать `response` как локальную переменную, он может быть уничтожен до завершения `async_write`. Поэтому ответ помещается в `std::shared_ptr` и захватывается в обработчик записи. | Ответ живёт до конца асинхронной записи |
