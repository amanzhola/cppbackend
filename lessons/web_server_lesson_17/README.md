## Теория

| 🧠 Теоретическая идея корректного завершения асинхронного сервера                                                                                        | 📖 Полное объяснение проблемы и способа её решения в Boost.Asio                                                                                                                                                    | ⚙️ Что именно используется в коде                                                                                                              | 🔥 Почему это важно для реальной асинхронной архитектуры                                        | ✅ Итог                                                           |
| -------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| В обычной синхронной серверной программе завершение через `Ctrl+C` может приводить к грубому останову процесса без возможности выполнить завершающий код | В асинхронной серверной архитектуре это плохо, потому что приложение может не успеть освободить ресурсы, закрыть соединения, записать финальные логи, удалить временные файлы и завершить рабочие потоки корректно | Для решения этой задачи в Boost.Asio используется `boost::asio::signal_set`, который позволяет асинхронно ожидать сигналы операционной системы | Без этого сервер может завершиться слишком грубо и оставить систему в неконсистентном состоянии | Для корректного shutdown нужен `signal_set` и вызов `ioc.stop()` |

## Сигналы

| 📌 Какие сигналы интересуют нас в этом уроке и как они используются для корректной остановки приложения | 📖 Подробное объяснение назначения каждого сигнала и реакции программы на него                                                                                        | 🧠 Что происходит после срабатывания обработчика                                                                                                                                             | 🔥 Что это даёт приложению на практике                                      | ✅ Итог                                                            |
| ------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| `SIGINT` и `SIGTERM`                                                                                    | В этом уроке нас интересуют два сигнала: `SIGINT` — interrupt signal, обычно приходит при `Ctrl+C`; `SIGTERM` — terminate signal, обычно отправляется командой `kill` | Когда приходит один из этих сигналов, в обработчике вызывается `ioc.stop();`, после чего цикл `ioc.run()` прекращает обработку асинхронных операций, и программа может завершиться корректно | Это даёт возможность закончить работу аккуратно, а не быть резко оборванной | Сервер умеет останавливаться правильно                            |
| `SIGKILL` и `SIGSTOP`                                                                                   | Из распространённых сигналов Linux два сигнала нельзя: поймать, заблокировать, проигнорировать. Это: `SIGKILL` и `SIGSTOP`                                            | Именно поэтому для корректного завершения обычно используют `SIGINT` и `SIGTERM`                                                                                                             | Нужно понимать границы того, что сервер может обработать сам                | Выбираются только сигналы, которые действительно можно обработать |

## Что делает demo1 по шагам

| 🧠 Шаг выполнения demo1              | 📖 Подробное объяснение действия на этом шаге                                                                                                               | ⚙️ Как это выглядит в коде              | ✅ Итог                                        |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- | --------------------------------------------- |
| 1. Определяет число рабочих потоков  | Используется `std::thread::hardware_concurrency()`. Это позволяет выбрать разумное количество потоков для обработки `io_context`.                           | Вычисляется число workers               | Подготовка пула потоков                       |
| 2. Создаёт io_context                | `net::io_context ioc(...)`. Это центральный объект Boost.Asio для обслуживания асинхронных операций.                                                        | Создаётся основной executor-контекст    | База всей async-логики                        |
| 3. Создаёт signal_set                | `net::signal_set signals(ioc, SIGINT, SIGTERM);`. Теперь приложение подписано на два сигнала: `SIGINT`, `SIGTERM`                                           | Настраивается ожидание сигналов         | Сервер начинает слушать сигналы               |
| 4. Запускает signals.async_wait(...) | Когда приходит сигнал, вызывается обработчик: `ioc.stop();`. Именно он останавливает цикл обработки асинхронных задач.                                      | Регистрируется async handler            | Будущий shutdown готов                        |
| 5. Создаёт таймер на 30 секунд       | Это нужно, чтобы показать: если сигнал не приходит, сервер продолжает жить; если сигнал приходит раньше, `io_context` останавливается до истечения таймера. | Появляется фоновая асинхронная операция | Видно поведение “ждём или завершаемся раньше” |
| 6. Запускает worker-потоки           | Через `RunWorkers(...)` вызывается: `ioc.run();`                                                                                                            | Стартует обработка событий              | Сервер реально начинает работать              |
| 7. После завершения run()            | Печатается: `Shutting down`. Это и есть признак корректного завершения.                                                                                     | Финальное сообщение после выхода из run | Graceful shutdown подтверждён                 |

## Важное замечание

| ⚠️ Что будет, если вызвать `acceptor.async_accept` без параметра executor                                                                 | 📖 Полное объяснение                                                                                                      | 🧠 Почему это важно для производительности                          | ✅ Итог                                                       |
| ----------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------ |
| Если вызвать `acceptor.async_accept` без параметра executor, функционально сервер будет работать, но производительность может уменьшиться | Потому что сокеты будут использовать тот же `strand`, что и `acceptor`, и часть работы станет выполняться последовательно | Корректность не пострадает, но сервер станет менее производительным | Лучше передавать отдельный executor/strand для новых сокетов |

# Почему этот код соответствует теории

| 🧠 Пункт соответствия кода теоретическому каркасу урока | 📖 Полный текст пояснения без потери содержания                                                                                                                                                                                                                                                                                                                                                                                                                                                               | ✅ Итог                                    |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| Пункт 1 — шаблонный Listener                            | Вот это реализовано прямо: `template <typename RequestHandler> class Listener : public std::enable_shared_from_this<Listener<RequestHandler>>`. И внутри есть именно те поля, которые даны в теории: ссылка на `io_context`, `tcp::acceptor`, `RequestHandler`. То есть: `net::io_context& ioc_;`, `tcp::acceptor acceptor_;`, `RequestHandler request_handler_;`. Это соответствует теоретическому каркасу урока.                                                                                            | Listener соответствует теории             |
| Пункт 2 — async_accept с отдельным executor             | Вот это ключевая строка: `acceptor_.async_accept( net::make_strand(ioc_), beast::bind_front_handler(&Listener::OnAccept, this->shared_from_this()));`. Что здесь важно: мы передаём новый strand для socket, а не используем молча strand самого acceptor. Это значит: acceptor живёт в своём последовательном исполнителе; каждый новый socket получает свой отдельный последовательный исполнитель; обработчики разных сокетов могут обслуживаться независимо. Именно это и даёт лучшую производительность. | Executor выбран правильно                 |
| Пункт 3 — bind_front_handler                            | В теории был пример: `auto handler = boost::beast::bind_front_handler(SomeFunc, arg1, arg2); handler(arg3, arg4, arg5); // Вызовет SomeFunc(arg1, arg2, arg3, arg4, arg5);`. В коде это используется так: `beast::bind_front_handler(&Listener::OnAccept, this->shared_from_this())`. То есть фактически создаётся обработчик, который потом будет вызван как: `OnAccept(self, ec, socket)`, где `self` уже заранее привязан.                                                                          | bind_front_handler используется по теории |
| Пункт 4 — enable_shared_from_this                       | Реализована именно по теории: наследование от `std::enable_shared_from_this`, привязка `shared_ptr` к обработчику. Это тоже сделано: `class Listener : public std::enable_shared_from_this<Listener<RequestHandler>>` и `beast::bind_front_handler(&Listener::OnAccept, this->shared_from_this())`. Зачем: чтобы объект Listener не был уничтожен раньше, чем завершится async_accept. Это классический паттерн для асинхронного Boost.Asio/Beast-кода.                                         | Lifetime management реализован корректно  |
